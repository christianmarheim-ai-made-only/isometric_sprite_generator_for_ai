# ADR-0008: Source Assets Require Separate Equipment and Rig-Bound Hit Proxies

- Status: Proposed
- Date: 2026-06-04
- Blocks: M3 and M4
- Related: ADR-0006 visible surface masks, ADR-0007 world metrics, ADR-0016 AI generation rollout

## Context

AI-generated 3D meshes often fuse hands, weapons, shields, sleeves, straps, and props. Fused geometry makes it hard to assign regions, generate stable masks, place sockets, or measure body metrics. The visual mesh can be messy and still render well at sprite scale; the gameplay surface cannot be ambiguous.

## Decision

From M3 onward, production source assets must separate visuals, equipment, sockets, and hit proxies.

Required source structure for humanoids:

```text
CHARACTER_ROOT
  ARMATURE
  VISUAL_BODY
  VISUAL_EQUIPMENT
    MAIN_HAND_WEAPON
    OFF_HAND_SHIELD
    GEAR
  HIT_PROXIES
    hit_head
    hit_torso
    hit_arm_l
    hit_arm_r
    hit_leg_l
    hit_leg_r
    hit_weapon
    hit_shield
    hit_gear
  METRIC_PROXIES
    metric_body
  SOCKETS
    origin
    head_center
    hand_l
    hand_r
    weapon_grip
    weapon_tip
    muzzle
    muzzle_back
    shield_center
```

The hitmask is generated from rig-bound hit proxy geometry, not directly from arbitrary AI visual mesh topology. Visual meshes may differ from proxy meshes as long as the resulting hit surface is believable.

Hard rejection cases:

- weapon fused into hand without separate weapon object/proxy
- shield fused into torso/arm without separate shield object/proxy
- no required sockets for the equipment class
- no arm proxies for humanoids
- weapon/shield/gear included in body metric proxy
- source scale unknown
- locomotion loop has root XY motion beyond tolerance

## Consequences

### Positive

- Makes AI-generated visuals usable because gameplay geometry is normalized.
- Masks, sockets, and metrics come from simple deterministic proxies.
- Human review focuses on proxy believability instead of topology minutiae.

### Negative

- Adds cleanup work per generated asset.
- Requires Blender-side source validation before export.
- Some generated meshes will be rejected even if they look good.

## Validation requirements

A source validator must run before Stage 4 render/export and check required collections, sockets, proxy objects, source scale, and root-motion tolerance.

## M1/M2 assumption

The arrow pilot is generated by Python and has no Blender source/proxy file. It is explicitly exempt from M3 source-asset rules.

## M3 review questions

- Which exact naming convention will the Blender exporter require?
- Do non-humanoid creatures need a separate proxy taxonomy?
- What cleanup budget per variant is acceptable before rejecting an AI source?
